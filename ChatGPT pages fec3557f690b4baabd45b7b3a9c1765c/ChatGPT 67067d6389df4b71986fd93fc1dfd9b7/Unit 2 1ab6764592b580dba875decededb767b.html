Q1. Explain types of software metrics in detail.

Software Metrics are quantitative measures that help evaluate software quality, efficiency, and performance.

Types of Software Metrics:

  1. Process Metrics:
    • Measure characteristics of the software development process.
    • Examples: Defect density, review efficiency, cycle time.
  1. Product Metrics:
    • Measure attributes of the software product itself.
    • Examples: Code complexity, maintainability, testability.
  1. Project Metrics:
    • Measure aspects related to software project management.
    • Examples: Cost estimation, schedule variance, effort estimation.
  1. Size Metrics:
    • Determine software size using Lines of Code (LOC) or Function Point (FP).
    • Example: "A program has 5000 LOC".
  1. Complexity Metrics:
    • Measure the logical complexity of code.
    • Example: Cyclomatic Complexity (higher complexity increases testing effort).
  1. Defect Metrics:
    • Track software defect rates during testing.
    • Examples: Defect density, Mean Time to Failure (MTTF).

Statement Coverage Testing for Given Code:

Code:

Read A
Read B
if A > B
    Print "A is greater than B"
else
    Print "B is greater than A"
endif

Test Cases for Statement Coverage:

  1. If A = 4, B = 2 → Executes both the if condition and print statement for "A is greater than B".
  1. If A = 4, B = -8 → Executes the same path as above (same if condition).
  1. If A = 2, B = 4 → Executes the else condition and prints "B is greater than A".

Statement Coverage Calculation:

  • Total statements: 4 (Read A, Read B, If-Else condition, Print statement).
  • Covered statements: All statements are executed at least once → 100% Statement Coverage.

Q3. What is System Testing and Acceptance Testing?

System Testing:

  • A level of testing where the entire integrated system is tested.
  • Ensures that the system meets functional and non-functional requirements.
  • Example: Checking if an e-commerce website can handle 1000 users at once.

Types of System Testing:

  1. Functional Testing – Ensures all features work as expected.
  1. Performance Testing – Checks system speed and responsiveness.
  1. Security Testing – Identifies security vulnerabilities.

Acceptance Testing:

  • Conducted to validate whether the software is ready for deployment.
  • Ensures it meets business needs and customer expectations.
  • Types of Acceptance Testing:
    1. User Acceptance Testing (UAT) – Conducted by end-users.
    1. Alpha Testing – Done in a controlled environment before release.
    1. Beta Testing – Done by real users before final launch.

Q4. What is Boundary Value Analysis? Explain with an example.

Boundary Value Analysis (BVA):

  • A Black Box Testing technique that tests boundary values of input ranges.
  • Errors often occur at boundary values rather than middle values.

Example: Suppose a login system accepts passwords of 6 to 12 characters.

  • Valid Inputs: 6, 7, 11, 12 characters.
  • Invalid Inputs: 5, 13 characters.
Test CaseInput ValueExpected Output
Minimum Boundary (Valid)6 charactersAccepted
Below Minimum (Invalid)5 charactersRejected
Maximum Boundary (Valid)12 charactersAccepted
Above Maximum (Invalid)13 charactersRejected

Q5. Why are test cases prioritized? What are the Priorities given to test cases?

Why Test Cases are Prioritized?

  • To focus on critical functionalities first.
  • To identify high-risk defects early.
  • To optimize testing time when resources are limited.
  • To ensure that most important features work as expected.

Test Case Priorities:

  1. High Priority: Critical functions (e.g., Login, Payment Processing).
  1. Medium Priority: Important but non-critical features (e.g., User Profile Update).
  1. Low Priority: UI enhancements, minor bugs.

Q6. Explain Integration Testing. Describe the need for Stub and Driver in Unit Testing.

Integration Testing:

  • Testing performed after unit testing to check how modules interact.
  • Ensures smooth data flow between components.

Types of Integration Testing:

  1. Top-Down Approach: Start testing from main modules to submodules.
  1. Bottom-Up Approach: Test submodules first, then integrate them.
  1. Big Bang Approach: Integrate and test all modules at once.

Need for Stub and Driver in Unit Testing:

  • Stub: A dummy module that simulates a missing component. Used in Top-Down Testing.
  • Driver: A temporary program that calls lower-level modules. Used in Bottom-Up Testing.

Q7. What are the testing fundamentals?

Testing Fundamentals:

  1. Testing Shows the Presence of Defects, Not Their Absence – Finding no bugs does not mean the software is perfect.
  1. Exhaustive Testing is Impossible – Testing all inputs is unrealistic.
  1. Defect Clustering – Most defects are found in few modules.
  1. Pesticide Paradox – Repeating the same tests reduces effectiveness.
  1. Testing is Context Dependent – Different software requires different testing techniques.
  1. Absence of Errors Fallacy – A defect-free software can still fail if it does not meet requirements.

Q8. Explain Statement Coverage Testing and Decision Coverage Testing.

Statement Coverage Testing:

  • Ensures every line of code is executed at least once.
  • Example: If a program has 10 lines, all lines should be tested at least once.

Decision Coverage Testing:

  • Ensures that all possible outcomes of decision points (if-else, switch) are tested.
  • Example:
    if (A > B)
    print "A is greater";
else
    print "B is greater";
    • Statement Coverage: Executes all lines at least once.
    • Decision Coverage: Tests both A > B and A ≤ B conditions.

Q9. What is System Testing? What are the types of System Testing?

System Testing:

  • Ensures that the complete system works as expected.
  • Validates both functional and non-functional aspects.

Types of System Testing:

  1. Functional Testing – Ensures all functionalities work correctly.
  1. Performance Testing – Measures speed, scalability, and responsiveness.
  1. Security Testing – Identifies vulnerabilities and security risks.
  1. Usability Testing – Checks how user-friendly the software is.
  1. Compatibility Testing – Ensures the system works across different environments.
